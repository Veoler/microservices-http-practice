# Решение задачи: HTTP-коммуникация между микросервисами (Coffee Shop)

Проект представляет собой учебную систему автоматизации кофейни, построенную на микросервисной архитектуре. Система состоит из трех независимых компонентов, взаимодействующих между собой по протоколу HTTP, и двух изолированных баз данных PostgreSQL.

## 🏗 Архитектура системы

```
┌──────────────────────────────────────────────────────────────┐
│                           Клиент                             │
│                      (Postman / curl)                        │
└──────────────────────────┬───────────────────────────────────┘
                           │
                           ▼
┌──────────────────────────────────────────────────────────────┐
│                    API Gateway (:8080)                       │
│                    Единая точка входа                        │
└───────────────┬─────────────────────────┬────────────────────┘
                │                         │
                ▼                         ▼
┌───────────────────────┐     ┌───────────────────────┐
│     Menu Service      │     │    Orders Service     │
│        :8081          │◀────│         :8082             │
│                       │     │                       │
│  Хранение напитков    │     │  Оформление заказов   │
│                       │     │                       │
│   PostgreSQL (menu_db)│     │ PostgreSQL (orders_db)│
└───────────────────────┘     └───────────────────────┘

```

### Компоненты:

1. **API Gateway (`:8080`)**: Единая точка входа. Проксирует запросы от клиента к соответствующим внутренним сервисам. Клиент "не знает" порты внутренней инфраструктуры.
2. **Menu Service (`:8081`)**: Управляет справочником напитков (название, цена, наличие на складе). Работает со своей БД `menu_db`.
3. **Orders Service (`:8082`)**: Отвечает за создание и вывод заказов. В момент создания заказа обращается к *Menu Service* по HTTP для синхронизации данных о напитке и проверки его наличия. Работает со своей БД `orders_db`.

* **Language:** Go (Golang)
* **Framework:** Gin Web Framework (`github.com/gin-gonic/gin`)
* **ORM:** GORM (`gorm.io/gorm`)
* **Database:** PostgreSQL (2 изолированные БД)
* **HTTP Client:** Стандартный пакет `net/http`

## 🚀 Быстрый старт

### 1. Подготовка баз данных

Создайте две независимые базы данных в вашей PostgreSQL (через psql или pgAdmin):

```sql
CREATE DATABASE menu_db;
CREATE DATABASE orders_db;

```

*Примечание: По умолчанию в коде используется DSN: `host=localhost user=postgres password=postgres ... sslmode=disable`. При необходимости измените креды в файлах `main.go` каждого сервиса.*

### 2. Запуск сервисов

Для запуска системы вам понадобятся **три терминала**:

**Терминал 1 (Menu Service):**

```bash
cd menu-service
go run main.go

```

**Терминал 2 (Orders Service):**

```bash
cd orders-service
go run main.go

```

**Терминал 3 (API Gateway):**

```bash
cd gateway
go run main.go

```

## 📡 Спецификация API (Эндпоинты Gateway)

Все запросы отправляются строго на порт Gateway (`:8080`).

### Меню (Menu Service)

* **`POST /api/drinks`** — Добавить напиток в меню.
* **`GET /api/drinks`** — Получить все напитки.
* **`GET /api/drinks/:id`** — Получить напиток по ID.

### Заказы (Orders Service)

* **`POST /api/orders`** — Создать новый заказ.
* **`GET /api/orders`** — Получить список всех заказов.
* **`GET /api/orders/:id`** — Получить заказ по ID.

## 💻 Примеры тестирования (CURL)

### 1. Наполнение меню

```bash
# Добавляем Капучино (в наличии)
curl -X POST http://localhost:8080/api/drinks \
  -H "Content-Type: application/json" \
  -d '{"name": "Капучино", "price": 350, "in_stock": true}'

# Добавляем Раф (нет в наличии)
curl -X POST http://localhost:8080/api/drinks \
  -H "Content-Type: application/json" \
  -d '{"name": "Раф", "price": 400, "in_stock": false}'

```

### 2. Создание заказа (Успешно)

```bash
curl -X POST http://localhost:8080/api/orders \
  -H "Content-Type: application/json" \
  -d '{"drink_id": 1, "quantity": 2}'

```

### 3. Проверка валидации и обработки ошибок

```bash
# Попытка заказать напиток, которого нет в наличии (Раф)
curl -X POST http://localhost:8080/api/orders \
  -H "Content-Type: application/json" \
  -d '{"drink_id": 2, "quantity": 1}'
# Ответ: 400 Bad Request {"error":"Напитка нет в наличии"}

# Попытка заказать несуществующий drink_id
curl -X POST http://localhost:8080/api/orders \
  -H "Content-Type: application/json" \
  -d '{"drink_id": 999, "quantity": 1}'
# Ответ: 400 Bad Request {"error":"Напиток не найден"}

```

### 4. Проверка устойчивости к отказам (Fault Tolerance)

Если остановить **Menu Service** (`Ctrl+C` в Терминале 1) и попробовать сделать заказ:

```bash
curl -X POST http://localhost:8080/api/orders \
  -H "Content-Type: application/json" \
  -d '{"drink_id": 1, "quantity": 1}'
# Ответ: 502 Bad Gateway {"error":"Menu Service недоступен: ..."}

```

## 🧠 Ответы на вопросы для самопроверки

**1. Почему Orders Service хранит `drink_name` и `price` в своей базе, а не запрашивает их каждый раз из Menu Service?**

* **Историчность данных (Снапшот):** Цена и название продукта могут измениться в будущем. Если пользователь купил кофе за 350 рублей, то в истории заказов должна остаться именно эта цена, даже если завтра позиция подорожает до 400.
* **Производительность:** При получении списка заказов (`GET /orders`) не нужно делать N дополнительных HTTP-запросов (проблема N+1) в Menu Service ради отображения текстовых полей.

**2. Что произойдёт, если Menu Service изменит цену напитка после создания заказа?**

* Ничего критического не произойдет. В таблице заказов (`orders_db`) останется зафиксированная на момент покупки цена. Это правильное бизнес-поведение для e-commerce систем.

**3. Зачем нужен API Gateway? Можно ли обойтись без него?**

* Обойтись можно, но с ним архитектура становится чище. Gateway предоставляет клиенту **единую точку входа** (один хост/порт, один контракт по CORS и авторизации). Он инкапсулирует внутреннюю топологию сети — нам не нужно открывать порты всех микросервисов наружу, что повышает безопасность. Также на уровне Gateway удобно делать сквозное логирование, rate-limiting и маршрутизацию.

**4. Как бы ты реализовал авторизацию в этой архитектуре? В каком сервисе она должна быть?**

* Авторизацию (проверку JWT-токенов / сессий) правильнее всего вынести на **API Gateway**. Он проверяет валидность токена на входе. Если токен валиден, Gateway обогащает заголовки запроса (например, прокидывает `X-User-Id: 42`) и передает запрос дальше внутренним сервисам, которым уже не нужно повторно ходить в базу пользователей для базовой проверки аутентификации.

**5. Что произойдёт, если во время создания заказа Menu Service станет недоступен после проверки наличия, но до сохранения заказа?**

* В текущей реализации Orders Service успешно сохранит заказ в свою БД, так как проверка уже пройдена, а данные (`drink`) сохранены в локальную переменную в памяти. В реальных высоконагруженных распределенных системах для решения таких проблем (когда состояние одной системы изменилось, а вторая об этом не знает) применяют паттерны распределенных транзакций (например, **Saga** или **Two-Phase Commit**), либо компенсирующие события через брокеры сообщений (Kafka/RabbitMQ).
